## 💡 Mesha Codebase Conventions and Gotchas
This guide highlights high-signal facts, operational constraints, and architectural patterns to help agents avoid common pitfalls in the codebase.#\n\n### ⚙️ Workflow & Development Quirks\n*   **Process Flow:** The application separates REST API logic from scheduled background work. The `SessionPollingScheduler` (in the worker) handles scheduled jobs; standard business logic lives in the service layer and is triggered synchronously by HTTP requests.\n*   **Flyway Migrations:** Never modify existing database migration files (`V*.sql`). Always add a new file: `V{n+1}__description.sql`. \n*   **State Management:** Frontend uses **TanStack Query (React Query)** for server state (in custom hooks) and **Zustand** for global client configuration/state.\n\n### 🔗 API & Data Access Contracts\n*   **Authentication:** Most API endpoints require the `Authorization: Bearer <clerk-jwt>`. The following are exceptions:\n    *   Webhooks (`/api/webhooks/*`)\n    *   Connector Agent claims (dedicated connector token auth)\n*   **Rate Limiting/Retries:** When communicating with external services (e.g., Blocks API, GitHub), handle exponential backoff and retries as outlined in the `SessionPollService` logic.\n*   **Pagination:** All list endpoints use a standardized `PagedResponse<T>` format: `{ content, page, size, totalElements, totalPages, last }`.\n\n### 🧱 Architecture & Boundaries\n*   **Core Worker Logic Location:** Background scheduling and session polling happen in the worker service at: `platform/backend-api/src/main/java/com/mesha/api/worker/`. This is where scheduled jobs must run.\n*   **Code Separation Layering:** The API uses a clear separation: Controllers for endpoints, Services for business logic, and DTOs for request/response data. Adhere to this `Controller <-> Service <-> Model` pattern.\n*   **Frontend Abstraction:** All REST calls from the frontend must go through `platform/frontend/src/lib/api-client.ts`, which handles auth tokens and correlation IDs automatically.\n\n### 📈 CICD & Testing (Non-obvious)\n*   **Testing Flow:** A full validation round requires: `npm run lint` $\rightarrow$ `npm run typecheck` $\rightarrow$ `npm run test`. Running them out of order or skipping steps can lead to invalid code.\n*   **Deployment Configs:** Frontends deploy to Vercel; Backends use Render (Docker container via `render.yaml`). A successful deployment always involves passing through the CI workflow defined in `.github/workflows/`."

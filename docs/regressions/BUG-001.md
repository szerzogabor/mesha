# Bug

Duplicate `github_pull_requests` records were created when GitHub retried webhook delivery. If GitHub sent the same `pull_request.opened` event more than once (which it does on network errors), the webhook handler inserted a new row each time, resulting in multiple PR records linked to the same `BlocksSession` and `pr_number`.

# Root Cause

The `github_pull_requests` table had no uniqueness constraint on `(session_id, pr_number)`. The webhook handler called `GitHubPullRequestService.save()` unconditionally, which always inserted a new row.

# Fix

Added a unique constraint on `(session_id, pr_number)` in Flyway migration `V37__deduplicate_github_pull_requests.sql`. The service layer was updated to use an upsert pattern (find-or-create) rather than unconditional insert.

# Notes

GitHub's webhook delivery guarantee is "at least once". All webhook handlers must be idempotent. The `github_webhook_events` table logs `delivery_id` to assist with debugging duplicate deliveries.

# Bug

Duplicate Blocks messages were inserted into `blocks_messages` when the backend worker restarted mid-poll or when a polling cycle overlapped. The same messages from the Blocks API were re-inserted because the system had no way to know which messages had already been persisted.

# Root Cause

`blocks_sessions` had no record of which messages had already been fetched from the Blocks API. On each poll, all messages returned by `BlocksAdapter.pollSession()` were inserted, resulting in duplicates when the worker restarted or a poll was retried.

# Fix

Added `api_message_offset` column to `blocks_sessions` in Flyway migration `V14__add_api_message_offset.sql`. The column tracks the highest message offset already persisted. `SessionPollTransactions` now only inserts messages with offsets strictly greater than the stored offset.

# Regression Test

`platform/backend-api/src/test/java/com/mesha/api/worker/SessionPollTransactionsTokenLimitTest.java`
Verifies that messages already at or below the stored offset are not re-inserted.

# Notes

The `api_message_offset` field must always be updated atomically with the messages insertion. Any refactoring of `SessionPollTransactions` must preserve this ordering guarantee.

# Project Management

## Purpose

The core domain of Mesha: organizing work into workspaces, projects, and issues. Supports custom statuses, labels, priorities, assignees, issue links, comments, and an immutable activity changelog.

---

## Responsibilities

- CRUD for workspaces, projects, issues, comments, and labels
- Custom project status management (ordered, named, colored)
- Issue priority management (`LOW`, `MEDIUM`, `HIGH`, `URGENT`)
- Assignee management (human users and AI agent definitions)
- Label management (workspace-scoped, M2M with issues)
- Issue link management (relationships between issues)
- Immutable activity event log (all state changes recorded)
- Readable issue identifiers (project key + sequential number)
- Real-time issue updates via Server-Sent Events (SSE)

## Public Interfaces

### REST Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET/POST` | `/api/workspaces` | List / create workspaces |
| `GET/POST` | `/api/workspaces/{id}/projects` | List / create projects |
| `GET/POST` | `/api/projects/{id}/issues` | List / create issues (paginated) |
| `GET` | `/api/projects/{id}/issues/stream` | SSE stream for real-time updates |
| `GET/PATCH/DELETE` | `/api/projects/{id}/issues/{issueId}` | Get / update / delete issue |
| `GET` | `/api/projects/{id}/issues/{issueId}/activity` | Issue activity log |
| `GET/POST` | `/api/issues/{id}/comments` | Issue comments |
| `GET/POST` | `/api/workspaces/{id}/labels` | Workspace labels |
| `GET/POST` | `/api/projects/{id}/statuses` | Custom project statuses |
| `GET/POST` | `/api/issues/{id}/links` | Issue links |

### Response Format
All list endpoints return `PagedResponse<T>`:
```json
{
  "content": [...],
  "page": 0,
  "size": 20,
  "totalElements": 100,
  "totalPages": 5,
  "last": false
}
```

## Important Business Rules

1. **Readable IDs:** Issues have human-readable identifiers (`{PROJECT_KEY}-{number}`, e.g., `TP-42`) generated by a sequence per project. Never reuse numbers.

2. **Custom statuses:** Each project defines its own ordered list of statuses. Issues reference status by name string. Status order is preserved for Kanban column ordering.

3. **Priority enum:** `LOW`, `MEDIUM`, `HIGH`, `URGENT` — stored as enum in DB.

4. **Activity log immutability:** `activity_events` rows are never updated or deleted. Every state change (status, priority, label, title, description, assignee) generates a new event row.

5. **Issue link types:** `BLOCKS`, `BLOCKED_BY`, `RELATES_TO`, `DUPLICATES`, `DUPLICATED_BY` — bidirectional relationships are stored as two rows.

6. **Workspace-scoped labels:** Labels belong to a workspace, not a project. Multiple projects share the same label pool.

7. **SSE stream:** The `/api/projects/{id}/issues/stream` endpoint pushes real-time updates to the frontend when issues in the project change. Only use Spring's `SseEmitter` — no third-party websocket libraries.

8. **Agent assignee:** An issue can have a human `assignee_id` and separately an `agent_assignee_id` referencing an `agent_definitions` row. These are independent.

9. **Automation on state change:** Every status transition must call `AutomationService.evaluate(STATUS_CHANGED, ...)` after persisting the change.

---

## Components That Must Not Be Modified Casually

- `ActivityService.log()` — adding conditions or skipping calls breaks the audit trail
- `IssueService.transitionStatus()` — must always fire `AutomationService` after the DB write
- `project_statuses` ordering logic — changing this breaks Kanban column order
- Issue ID generation sequence — resetting or modifying creates duplicate readable IDs

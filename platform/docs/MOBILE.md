# Mesha Mobile & PWA Guide

Mesha is an **installable, Android-first Progressive Web App** built entirely
inside the existing Next.js frontend (`platform/frontend`). There is **no
separate mobile app, no React Native, and no second frontend** — the mobile
experience is the same codebase, made responsive and installable.

This document covers how the mobile experience is built, how to install it, how
to test it, and the roadmap for push notifications.

## Install Mesha on Android

**From an Android phone (Chrome):**

1. Open the deployed Mesha URL (or visit **`/download`**).
2. Tap **Install app**, or open Chrome's **⋮** menu → **Install app / Add to Home screen**.
3. Confirm. Mesha appears on your home screen and launches full-screen
   (standalone), looking and behaving like a native Android app.

**From desktop:** visit `/download` and **scan the QR code** with your phone, then
follow the steps above.

The `/download` page also shows the current app version and release notes. The
landing page (`/`) has a prominent **"Download the App"** hero CTA.

## Architecture

| Concern | Where | Notes |
|---|---|---|
| Web App Manifest | `src/app/manifest.ts` | Next.js metadata route → `/manifest.webmanifest` |
| Icons | `public/icons/*`, `public/apple-touch-icon.png` | Generated by `scripts/generate-icons.mjs` |
| Service worker | `public/sw.js` | Offline app-shell, registered by `PwaProvider` |
| Offline fallback | `public/offline.html` | Branded standalone offline screen |
| Install + platform state | `src/context/PwaContext.tsx` | `usePwa()` → `canInstall`, `isStandalone`, `promptInstall()` |
| Install affordances | `src/components/pwa/*` | `InstallAppButton`, `InstallAppCard`, `QrCode` |
| Download page | `src/app/download/page.tsx` | Public route (see `middleware.ts`) |
| Landing page | `src/app/page.tsx` | Public; redirects signed-in users to `/workspaces` |
| Bottom navigation | `src/components/layout/MobileNav.tsx` | Mobile-only Android-style tab bar |
| Dashboard | `src/components/dashboard/WorkspaceDashboard.tsx` | Workspace home |
| Projects index | `src/app/workspaces/[workspaceId]/projects/page.tsx` | Destination for "Projects" tab |
| Settings hub | `src/app/workspaces/[workspaceId]/settings/page.tsx` | Destination for "Settings" tab |
| Mobile bottom sheet | `src/components/ui/BottomSheet.tsx` | Reusable touch action surface |
| Kanban move (touch) | `src/components/issues/MoveStatusSheet.tsx` | Tap-to-move status picker |
| Notifications foundation | `src/lib/notifications/*`, `src/hooks/useNotifications.ts` | Abstraction only |

**Design principle:** every mobile screen **reuses the existing hooks, API
client, types, and components**. New routes are thin shells over existing data
hooks — no duplicated pages or business logic.

## PWA internals

### Manifest (`src/app/manifest.ts`)
- `display: standalone`, `start_url: /workspaces`, brand `theme_color`.
- Icons include `192`/`512` raster icons plus **maskable** variants so Android
  masks the icon into the device's icon shape, and an SVG for crisp scaling.

### Service worker (`public/sw.js`)
- **Navigations:** network-first, falling back to `/offline.html` when offline
  — keeps live, auth-gated data fresh while still giving installed users a
  branded offline screen.
- **Static build assets** (`/_next/static/*`, `/icons/*`): cache-first (they're
  content-hashed and safe to cache immutably).
- **Everything else** (API, auth, telemetry, cross-origin): passthrough.
- Bump `CACHE_VERSION` to invalidate caches on deploy.

The service worker is registered client-side by `PwaProvider` after `load`.

### Viewport & theming (`src/app/layout.tsx`)
- `viewport-fit=cover` + `env(safe-area-inset-*)` utilities (`.pt-safe`,
  `.pb-safe`, `.pb-mobile-nav` in `globals.css`) handle notches and the Android
  gesture bar in standalone mode.
- `theme_color` adapts to light/dark.

### Install flow (`PwaContext`)
- Captures the Chromium `beforeinstallprompt` event so we can offer an in-app
  **Install** button (`InstallAppButton`) instead of relying on the browser menu.
- Falls back to `/download` instructions on iOS / browsers that don't fire it.
- Detects `display-mode: standalone` to hide install prompts once installed.

## Mobile navigation & layouts

- **Bottom tab bar** (`MobileNav`): `Home · Issues · Sessions · Projects ·
  Settings`. Mobile-only (`md:hidden`); the desktop sidebar is unchanged. Active
  tab is derived from the pathname; project-scoped tabs use the current /
  most-recently-visited / first project.
- The desktop **sidebar** remains the navigation on `md+` and continues to work
  as an overlay drawer on small screens (pre-existing behavior).
- Pages are mobile-first: cards/stacked rows on phones, grids/tables on `md+`.
  Verified with no horizontal overflow at **360 / 390 / 412 px**.

## Kanban on touch devices

Drag-and-drop is preserved on desktop. On phones, each Kanban card shows a
**Move** button (`md:hidden`) that opens a `MoveStatusSheet` bottom sheet to
pick a destination status. The move goes through the same handler as
drag-and-drop, so **ticket rules / restrictions are still enforced** (rule
violations surface the existing `RuleViolationDialog`).

## Push notifications (foundation)

> **Status: foundation only.** The architecture is in place; delivery is *not*
> implemented yet, per the project's phased plan.

- `src/lib/notifications/types.ts` — `NotificationEventType`
  (`SESSION_COMPLETED`, `SESSION_FAILED`, `PULL_REQUEST_CREATED`,
  `AGENT_OFFLINE`), `NotificationPayload`, and the `NotificationProvider`
  interface (mirrors the backend `ProviderAdapter` pattern).
- `src/lib/notifications/provider.ts` — a `NoopNotificationProvider` (default)
  that tracks browser permission but never delivers, plus a documented outline
  for a future `FirebaseCloudMessagingProvider`.
- `src/hooks/useNotifications.ts` — `permission`, `isEnabled`, and an `enable()`
  action ready to wire into UI.

**To finish FCM delivery (future work):**
1. Add the `firebase` dependency + `NEXT_PUBLIC_FIREBASE_*` config and a VAPID key.
2. Add `public/firebase-messaging-sw.js` and implement
   `FirebaseCloudMessagingProvider.registerDevice()` (obtain FCM token).
3. Backend: a `device_tokens` table + `POST /api/notifications/devices`, and
   fan-out that emits the four event types on the relevant domain transitions.
4. Swap the provider via `setNotificationProvider(...)`.

## Testing

Playwright specs (run with `npm run test:e2e` in `platform/frontend`):

| Spec | Coverage |
|---|---|
| `e2e/tests/pwa.spec.ts` | Manifest validity, SW/offline/icon reachability, PWA meta tags, landing CTA, download page, public-page mobile overflow, bottom-nav presence (auth), desktop nav hidden (auth) |
| `e2e/tests/mobile.spec.ts` | Sign-in + workspace/project responsiveness on iPhone 14 & Pixel 7 (pre-existing, still valid) |

Auth-gated tests `test.skip` themselves unless `E2E_TEST_EMAIL` /
`E2E_TEST_PASSWORD` are set. The mobile projects (`mobile-chrome` = Pixel 7,
`mobile-safari` = iPhone 14) are configured in `playwright.config.ts`.

Manual checks:
- Chrome DevTools → **Application → Manifest** (no install errors) and
  **Lighthouse → PWA** category.
- Toggle the **Pixel 7 / 360px** device toolbar and confirm no horizontal scroll.

## Regenerating app icons

Icons are produced from a dependency-free Node script (no ImageMagick/rsvg
needed), rasterizing the brand "M" mark with Node's built-in `zlib`:

```bash
cd platform/frontend
node scripts/generate-icons.mjs
```

Outputs `public/icons/icon-{192,512}.png`, the `*-maskable-*` variants, and
`public/apple-touch-icon.png`. Edit the brand color / glyph in the script (or
`public/icons/icon.svg` for the scalable source) and re-run.

## Follow-ups / known tech debt

- **QR code** (`src/components/pwa/QrCode.tsx`) currently renders via a public
  QR image endpoint with graceful fallback to the plain URL. Vendoring an
  offline QR generator (or generating the QR server-side) would remove the
  third-party runtime dependency.
- **Push notification delivery** is intentionally unimplemented (see above).
- Screenshots for this guide can be captured from the Pixel 7 Playwright project
  (`screenshot: "on"`) and added under `platform/docs/` when a deployed preview
  with Clerk credentials is available.
